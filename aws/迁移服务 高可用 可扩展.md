## 问题 #244（纠错后）

一家公司正在使用一个运行在 **8 个 Amazon EC2 实例**上的**内容管理系统（CMS）**。  
这些 EC2 实例同时包含 **Web 服务器和数据库软件**。  

公司必须：
- 使其**网站平台高度可用**
- 并且必须使网站**能够扩展以满足用户需求**

解决方案架构师应该推荐哪种方案来满足这些要求？

A. 将数据库迁移到 Amazon RDS 并启用自动备份。在同一可用区手动启动另一个 EC2 实例。在一个可用区中配置 Application Load Balancer，并将两个实例设置为目标。  
B. 将数据库迁移到与现有 EC2 实例在同一可用区的 Amazon Aurora，并在同一可用区创建一个只读副本。手动在同一可用区启动另一个 EC2 实例。配置 Application Load Balancer，并将两个 EC2 实例设置为目标。  
C. 将数据库迁移到 **另一个可用区的 Amazon Aurora**，并在该可用区创建一个只读副本。从 EC2 实例创建一个 **AMI**。在 **两个可用区**配置 Application Load Balancer，并附加一个使用该 AMI 的 **Auto Scaling 组**，跨两个可用区。  
D. 将数据库迁移到一个单独的 EC2 实例，并将备份计划到 Amazon S3。从原始 EC2 实例创建一个 AMI。在两个可用区配置 Application Load Balancer，并附加一个使用该 AMI 的 Auto Scaling 组。

---

## 正确答案
**👉 C**

---

## 解析

### 考点
- **Web 与数据库解耦**
- **Multi-AZ 高可用**
- **Auto Scaling 水平扩展**
- AWS 托管数据库优于自管

---

### 为什么选 C

**方案 C 做对了所有关键点：**

#### 1️⃣ 数据库层
- 使用 **Amazon Aurora**
  - AWS 托管
  - 高可用
  - 高性能
- 数据库部署在 **独立可用区**
- 创建只读副本 → 提升可用性与读扩展能力

#### 2️⃣ 应用层
- 从现有 EC2 创建 **AMI**
- 使用 **Auto Scaling 组**
- 跨 **两个可用区** 自动扩展

#### 3️⃣ 流量入口
- **Application Load Balancer**
- 跨可用区分发流量

👉 满足题目全部要求：
- 高可用（Multi-AZ）
- 可扩展（Auto Scaling）
- 架构合理、符合 AWS 最佳实践

---

### 其他选项为什么错

#### ❌ A. RDS + 单 AZ + 手动扩容
- 单可用区
- 无自动扩展
- 自动备份 ≠ 高可用

---

#### ❌ B. Aurora 但仍在单 AZ
- 同一可用区部署
- 只读副本不能解决 AZ 故障
- 手动扩容不具备弹性

---

#### ❌ D. 数据库仍在 EC2
- 自管数据库
- 运维成本高
- 无原生高可用
- 不是云原生架构

---

## 一句话记忆
> **高可用 + 可扩展网站 = ALB + Auto Scaling（Multi-AZ）+ 托管数据库**

考试秒杀版：

> **凡是“网站高可用 + 扩展” → 拆库、上 ASG、跨 AZ**
